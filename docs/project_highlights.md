# 族谱管理系统实验亮点整理

## 1. 核心功能亮点

本项目围绕“多用户、多族谱、大规模成员、深层亲缘关系查询”设计，实现了实验要求中的注册登录、权限隔离、Dashboard 统计、族谱与成员 CRUD、协作者邀请、成员模糊查询、树形预览、祖先递归查询、亲缘链路查询、CSV 分支导出和核心 SQL。

树形预览为 DFS 嵌套树：

- 先展示根祖先，再完整展开第一个子女的全部后代子树，然后再进入下一个子女分支。
- 节点卡片展示 ID、姓名、性别、世代、生卒年。
- 节点内展示配偶标签，树枝连线展示亲子层级，避免原来按世代平铺导致亲缘关系不清晰的问题。
- 大分支限制前 200 个渲染节点，防止 5 万人族谱页面卡死；完整分支可通过 CSV 导出。

## 2. 数据库设计与范式

核心表包括：

- `users`：注册用户。
- `genealogies`：族谱基本信息。
- `genealogy_collaborators`：用户参与族谱的多对多关系。
- `members`：族谱人物基本信息。
- `parent_child_relations`：父母与子女关系。
- `marriages`：婚姻关系。

设计原则是把独立多值事实拆成清晰的关系表，而不是在 `members` 中存储父亲、母亲、配偶列表、子女列表等重复字段。这样避免更新异常、插入异常和删除异常。

范式结论：

- 所有字段均为原子值，满足 1NF。
- 复合主键表的非主属性完全依赖整个主键，满足 2NF。
- 非主属性不依赖其他非主属性，满足 3NF。
- `users`、`genealogies`、`members`、`genealogy_collaborators`、`parent_child_relations`、`marriages` 的主要函数依赖决定因素均为候选键，可说明达到 BCNF。
- 配偶、亲子、协作者等多值依赖都已经分解为独立且有明确含义的关系表，可说明核心业务表达到 4NF。

## 3. 约束与数据质量

数据库层提供多重约束：

- 外键约束保证族谱、成员、亲子关系、婚姻关系不会出现孤立记录。
- `gender`、`status`、年份范围等 `CHECK` 约束保证枚举和时间合法。
- `death_year >= birth_year` 保证生卒年顺序。
- `member1_id < member2_id` 保证婚姻关系只有一个方向，避免重复。
- 每个孩子最多一个父亲、一个母亲，由部分唯一索引实现。
- 触发器保证父母和子女必须属于同一族谱。
- 触发器保证父母出生年早于子女。
- 触发器保证 father/mother 关系与性别匹配。
- 触发器用递归 CTE 防止亲子关系形成环。

这些约束使系统不只在前端校验，还能在数据库层抵御错误数据。

## 4. 核心查询逻辑

项目中最能体现数据库课程要求的查询包括：

- 配偶及所有子女查询：通过 `marriages` 和 `parent_child_relations` 定位家庭关系。
- 祖先查询：使用 Recursive CTE 从成员向上递归父母，输出所有历代祖先。
- 亲缘链路查询：把亲子边和婚姻边统一视为图边，再用递归 CTE 搜索两人之间的最短可达链路。
- 统计分析：按世代统计平均寿命，筛选高龄无配偶男性，找出早于本世代平均出生年的成员。
- 分支导出：按 DFS 顺序导出某根节点的完整后代分支，便于备份和验收。

## 5. 性能优化设计

索引覆盖了核心访问路径：

- `idx_members_genealogy_name`：族谱内姓名检索。
- `members_fts`：FTS5 全文索引，支持姓名/简介检索，触发器自动同步。
- `idx_parent_child_parent(parent_id, child_id)`：从父母查子女，支撑树形预览和后代查询。
- `idx_parent_child_child(child_id, parent_id)`：从子女查父母，支撑祖先查询。
- `idx_parent_child_child_type(child_id, relation_type, parent_id)`：快速定位父亲或母亲。
- `idx_members_genealogy_generation_birth`：支撑成员列表、世代排序和统计分析。
- `idx_members_life_stats`：支撑年龄、性别和寿命类统计。
- `idx_marriages_member1`、`idx_marriages_member2`：双向配偶查询。

树形预览也做了应用层优化：一次性读取当前族谱的成员、亲子边、婚姻边，构造哈希映射后 DFS 展开，避免对每个节点反复访问数据库造成 N+1 查询。

## 6. 存储、事务与并发控制

应用连接 SQLite 时启用：

- `foreign_keys = ON`：强制外键。
- `journal_mode = WAL`：提高读写并发能力。
- `synchronous = NORMAL`：在 WAL 下减少同步开销。
- `busy_timeout = 5000`：并发写入冲突时等待，减少锁错误。

写操作统一放在事务上下文中执行。成员、亲子关系、婚姻关系新增或更新时，如果任何约束失败，数据库会自动回滚本次写入，保证一致性。

## 7. 前端与验收展示亮点

界面强化了以下部分：

- 树形预览从缩进列表升级为带连线的族谱树。
- 人物节点中同时展示基础信息和配偶关系。
- 树容器限制高度并支持滚动，适合大族谱演示。
- 成员检索、查询中心、分支导出入口保持在同一详情页，演示路径短。
- Dashboard 显示每个可访问族谱的总人数和男女比例，体现多用户权限隔离后的统计视图。

验收建议演示顺序：

1. 用 `admin / admin123` 登录。
2. 进入示例族谱，搜索成员并编辑成员信息。
3. 输入根成员 ID 查看 DFS 树形预览，说明配偶标签和亲子连线。
4. 打开查询中心，演示祖先查询、亲缘链路查询、配偶及子女查询。
5. 展示 `schema.sql` 中约束、触发器、索引。
6. 展示 `docs/database_design.md` 的 ER、范式、索引和并发优化说明。
